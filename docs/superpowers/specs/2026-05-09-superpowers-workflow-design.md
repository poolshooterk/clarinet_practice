## 概要

expo-template における新機能追加・大きな変更を行う際の、superpowers スキルを活用した標準開発ワークフロー。

4 フェーズ (設計 → 計画 → 実装 → 完了) を順番に踏み、各フェーズで対応するスキルを呼び出す。

## 対象範囲

- **適用する**: 新しいルート、新しいフォーム、新しいストア、新しいコンポーネント群など、複数ファイルにまたがる変更
- **適用しない**: バグ修正、リファクタリング、依存関係の更新など小さな単発変更

## フェーズ定義

| フェーズ | スキル                           | 入口条件                     | 出口条件                                                            |
| -------- | -------------------------------- | ---------------------------- | ------------------------------------------------------------------- |
| 設計     | `brainstorming`                  | 機能の追加・変更要求がある   | 設計 spec が `docs/superpowers/specs/` に保存され、ユーザが承認済み |
| 計画     | `writing-plans`                  | 設計 spec が承認済み         | 実装計画が `docs/superpowers/plans/` に保存され、ユーザが承認済み   |
| 実装     | `test-driven-development`        | 実装計画が承認済み           | すべてのタスクが完了、品質チェック 4 ステップを通過済み             |
| 完了     | `finishing-a-development-branch` | 実装が品質チェックを通過済み | PR 作成 or ブランチがレビュー待ち状態                               |

## 各フェーズの詳細

### 設計フェーズ (`brainstorming`)

**目的**: 実装前に要件と設計を固め、手戻りを防ぐ。

- ユーザと対話しながら目的・制約・成功基準を明確化する
- 2〜3 の実装アプローチを提案してトレードオフを議論する
- 承認された設計を `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` に保存してコミットする

### 計画フェーズ (`writing-plans`)

**目的**: 設計を具体的な実装手順に分解する。

- 設計 spec を読み込み、ファイル単位・タスク単位の実装ステップを作成する
- 依存関係とリスクを明記する
- 承認された計画を `docs/superpowers/plans/YYYY-MM-DD-<topic>-plan.md` に保存してコミットする

### 実装フェーズ (`test-driven-development`)

**目的**: テストファーストで品質を担保しながら実装する。

- 計画のタスクを 1 件ずつ: テスト作成 → 実装 → グリーン確認 のサイクルで進める
- テスト戦略マトリクス (CLAUDE.md 参照) に従い unit / integration / E2E を使い分ける
- 全タスク完了後、品質チェック 4 ステップ (`lint / format:check / tsc / test`) を通過してから次フェーズへ

### 完了フェーズ (`finishing-a-development-branch`)

**目的**: ブランチを出荷可能な状態に仕上げる。

- コミット履歴・変更差分・残タスクを確認する
- PR を作成し、レビュー待ち状態にする
- コミット運用 (CLAUDE.md 参照) に従い、push 前にユーザへ確認する

## 設計上の決定

- **既存の「コード品質チェック」「コミット運用」との関係**: このワークフローはフェーズ間の順序を定める。コミットの詳細手順は既存セクションに委譲する
- **スキルは Skill ツール経由で呼び出す**: `superpowers:brainstorming` のような完全修飾名で指定する
- **小さな変更はフローをスキップしてよい**: YAGNI。形式に縛られず直接実装できる
